# ReplyDesk Lite

소상공인 매장이 고객 문의를 한 곳에서 접수·분류하고, AI 답변 초안·고객 이력·지식베이스·팀 권한·일일 리포트까지 관리하는 **상담 운영 웹앱**입니다.

## 프로젝트 목표

ReplyDesk Lite의 목표는 소규모 사업자가 흩어진 고객 문의를 놓치지 않고, 반복 답변과 상담 이력을 체계적으로 관리할 수 있는 가벼운 상담 CRM을 만드는 것입니다.

현재 구현은 범용 소상공인 상담 도구를 출발점으로 하지만, 장기 방향은 모든 업종을 넓게 지원하는 범용 CRM이 아니라 **특정 전문 직군에 깊게 맞춘 vertical CRM**입니다. 특히 타투이스트, 반영구/속눈썹/네일샵, 공방/원데이클래스, 출장 수리 기사처럼 상담 전에 확인해야 할 조건이 많고 견적·예약·주의사항 안내가 반복되는 업종을 우선 검토합니다.

제품이 해결하려는 핵심 문제는 다음과 같습니다.

- 고객 문의가 인스타 DM, 카카오톡, 네이버톡톡, 문자 등 여러 채널에 흩어져 상담 흐름이 끊기는 문제
- 가격, 예약금, 준비물, 환불, 주의사항처럼 반복되는 답변을 매번 새로 작성해야 하는 문제
- 상담 담당자가 바뀌거나 시간이 지나면 고객의 이전 문의 맥락을 다시 파악해야 하는 문제
- 문의가 실제 예약, 방문, 결제로 이어졌는지 작게라도 추적하기 어려운 문제
- 업종별 운영 정책과 답변 기준이 사람마다 달라 고객 경험이 흔들리는 문제

## 비즈니스 목표

이 프로젝트의 비즈니스 목표는 대기업형 CRM이나 범용 상담툴과 정면 경쟁하는 것이 아니라, 작은 전문 사업자가 바로 이해하고 비용을 지불할 수 있는 좁고 선명한 문제를 찾는 것입니다.

초기 수익화 가설은 다음과 같습니다.

- **타깃**: 1인~10인 규모의 전문 서비스 사업자 또는 소규모 팀
- **초기 유료 가치**: 문의 정리, 상담 이력, 업종별 지식베이스, AI 답변 초안, 예약 전환 관리
- **가격 방향**: 월 구독형 SaaS를 기본으로 하되, AI 사용량이 많은 기능은 사용량 제한 또는 상위 요금제로 분리
- **차별화 방향**: 범용 CRM 기능 수가 아니라 업종별 상담 흐름, 질문 템플릿, 답변 정책, 고객 상태 관리의 정확도
- **검증 방식**: 랜딩 페이지, 직군별 인터뷰, 실제 문의 샘플 입력 테스트, 유료 사용 의향 확인

현재 범용 소상공인 버전은 사업성 검증 결과 무료 대안과 비용 구조의 압박이 커서 그대로 확장하기 어렵다고 판단합니다. 따라서 다음 단계의 제품 방향은 **타투 상담 CRM 같은 단일 직군 특화 MVP**로 좁혀, 고객 문의가 실제 예약과 매출로 연결되는 흐름을 더 깊게 구현하는 것입니다.

## 기술 스택

- **프레임워크**: Next.js 15 (App Router) + React 19
- **언어**: TypeScript (strict)
- **데이터베이스**: PostgreSQL (`pg`)
- **AI**: OpenAI Responses API (답변 초안 생성)

## 빠른 시작

### 1. 요구사항

- Node.js 18+
- 로컬 PostgreSQL (기본 포트 5432)

### 2. 의존성 설치

```powershell
npm install
```

### 3. 환경 변수 설정

`.env.example`를 복사해 `.env.local`을 만들고 값을 채웁니다.

```powershell
DATABASE_URL="postgresql://replydesk_app:admin1234@127.0.0.1:5432/replydesk_lite"
OPENAI_API_KEY=sk-...
OPENAI_MODEL=gpt-5.5
```

`OPENAI_API_KEY`가 없으면 **AI 답변 생성만 비활성**으로 동작하고, 나머지 기능과 E2E 검증은 그대로 사용할 수 있습니다.

### 4. DB 스키마 적용

```powershell
npm run db:schema
```

### 5. 개발 서버 실행

```powershell
npm run dev -- --hostname 127.0.0.1 --port 4183
```

브라우저에서 `http://127.0.0.1:4183/`을 엽니다.

## 주요 기능

- 문의 접수, 자동 분류, 우선순위 판단, 상태 관리
- AI 답변 초안 생성 및 답변 품질 점검

- 일일 운영 리포트

## API 엔드포인트

모든 데이터 API는 세션 쿠키로 해석된 워크스페이스 컨텍스트 기준으로 동작합니다.

| 영역 | 메서드 · 경로 |
|------|------|
| 인증 | `POST /api/auth/login` · `POST /api/auth/register` · `POST /api/auth/logout` |
| 세션 | `GET /api/session` · `POST /api/session/select` |
| 문의 | `GET·POST·DELETE /api/inquiries` · `PATCH·DELETE /api/inquiries/[id]` |
| 고객 | `GET /api/customers` · `PATCH /api/customers/[id]` |
| 지식베이스 | `GET·PUT /api/knowledge` |
| 설정 | `GET·PUT /api/settings` |
| 멤버 | `GET·POST /api/members` · `PATCH·DELETE /api/members/[userId]` |
| 리포트 | `GET /api/report/daily` |
| AI | `POST /api/generate-reply` |
| 상태 | `GET /api/db/health` |

```text

                       (AuthScreen, InquiryList, DetailPanel, CustomerPanel,

  constants.ts         앱 상수와 업종 템플릿

  utils.ts             CSV 가져오기/내보내기 유틸

  schema.sql           PostgreSQL 스키마와 데모 데이터

  e2e-smoke.mjs        API 기반 smoke test (npm run test:e2e)

  index.ts             공용 타입
```

## 검증

```powershell
npm run typecheck   # 타입 검사 (tsc --noEmit)
npm run build       # 프로덕션 빌드
npm run test:e2e    # API 기반 E2E smoke test
```

> ⚠️ `next dev`가 실행 중일 때 `npm run build`를 같은 `.next`에 돌리면 캐시가 깨질 수 있습니다. 빌드는 dev 서버를 끈 상태에서 실행하세요.

E2E는 기본적으로 `http://127.0.0.1:4183`을 대상으로 합니다. 다른 주소를 쓰려면 `TEST_BASE_URL`을 지정합니다.

```powershell
$env:TEST_BASE_URL="http://127.0.0.1:3000"
npm run test:e2e
```

### 치명 가정 검증 결과

**가정 #1 — 소상공인의 지불 의향: 반박 우세**

- 채널톡 무료 플랜에서 카카오 상담톡, 네이버 톡톡, 인스타 DM 연동을 무제한 무료 제공
- 네이버 스마트플레이스(예약 관리), 카카오톡 채널(상담/알림)은 완전 무료이며 기본 인프라로 정착
- 소상공인은 고정비 압박 속에서 SaaS 구독 지불 저항이 매우 강함

**가정 #2 — 무료 경쟁 도구 대비 차별점: 반박 우세**

- "멀티채널 통합" → 채널톡 무료에서 이미 제공
- "AI 답변 초안" → 유일한 차별점이나, 소상공인 문의 대부분은 영업시간/가격/위치 등 단순 정형 질문
- 채널톡이 이미 ALF v2(AI 상담봇) 출시

**가정 #3 — 기능 규모 적합성: 반박**

- 1~3명 매장에서 5단계 권한(Owner/Admin/Lead/Agent/Viewer)은 과잉 설계
- 하루 5~20건 처리하는 매장에서 일일 리포트, CSV 관리는 불필요
- 타겟과 제품 사이 미스매치 존재

**가정 #4 — AI 비용 구조: 반박 우세**

- GPT-5.5 답변 1건당 약 50~200원, 월 300건 기준 약 3만원의 AI 비용
- 소상공인 수용 가격대(월 2~3만원)에서 AI 비용만으로 매출 초과 또는 마진 제로

### 검증 실험 (MVP)

1. **지불 의향 테스트** — 타겟 업종 소상공인 20곳 직접 인터뷰. 5곳 이상 즉시 수용이면 통과
2. **경쟁 도구 인지도 테스트** — 채널톡/카카오/네이버 무료 도구 이미 사용 중인지 확인
3. **랜딩 페이지 사전 등록** — AI 답변 초안 1가지만 강조, 타겟 광고 5만원 집행, 전환율 3% 이상이면 수요 신호

### 판정: NO-GO (현재 형태 기준)

1. **치명적 시장 미스매치** — 기능 복잡도는 중소기업 CS팀 수준이나 타겟은 1~3명 매장
2. **무료 대안이 너무 강함** — 채널톡 무료, 네이버/카카오 무료 인프라가 이미 기본 도구
3. **AI 차별점의 경제성 문제** — AI가 유일한 차별점이나 정형 문의에는 과잉, API 비용이 역마진 구조
4. **불공정 우위 부재** — 도메인 전문성, 유통 채널, 데이터 등 방어 가능한 경쟁 우위 없음

### 피벗 고려 방향

| 방향 | 설명 |
|------|------|
| 타겟 좁히기 | 상담이 복잡하고 객단가 높은 단일 업종(예: 병원/클리닉)에 집중 |
| 기능 극단적 단순화 | 5단계 권한/CSV/리포트 제거, "카카오 문의 → AI 초안" 하나에만 집중 |

| B2B2C | 프랜차이즈 본사 납품 → 가맹점 일괄 도입으로 개별 영업 비용 회피 |
